**7.1. COMPLETE LIST OF TRIGGERS (EVENT → ACTION)**
───────────────────────────────────────────────────────────────────────────────────

Triggers are the foundation of event‑driven automation. Each trigger defines which
event (data change, timer signal, parsing result) initiates which action (or chain
of actions). All triggers are implemented via Redis Streams (or equivalent) and are
processed by the respective agents. Below is a detailed list of all 16 triggers,
their conditions, actions, priorities, retry mechanisms, and error handling.

**7.1.2. TECHNICAL IMPLEMENTATION OF TRIGGERS**
───────────────────────────────────────────────────────────────────────────────────

All triggers are built on publishing events to Redis Streams. Each source (agent
or external service) publishes messages to a specific topic. Subscriber agents read
these topics and perform actions. For guaranteed delivery, consumer groups are used
with automatic acknowledgment (ack) after successful processing.

**Example Redis Streams configuration (Python):**
```python
import redis
import json

redis_client = redis.Redis(host='redis', port=6379, decode_responses=True)

# Publish event (e.g., from WB API monitor)
event = {
    "event_id": "uuid",
    "timestamp": "2026-03-28T10:00:00",
    "type": "stock_updated",
    "payload": {"sku": "PICKET-001", "stock": 120}
}
redis_client.xadd("stock_events", {"data": json.dumps(event)})

# Subscription by agent D3
def process_stock_events():
    last_id = "0"
    while True:
        events = redis_client.xread({"stock_events": last_id}, block=5000, count=10)
        for stream, messages in events:
            for message_id, fields in messages:
                event = json.loads(fields['data'])
                # process event
                process_stock_event(event)
                # acknowledge (if using consumer group)
                redis_client.xack("stock_events", "d3_group", message_id)
            last_id = message_id

            For fault tolerance, a Dead Letter Queue (DLQ) is used. If an event is not processed
after 3 attempts, it is moved to a separate topic dlq. The operator can view the
DLQ through the console and manually trigger reprocessing.

7.1.3. TRIGGER MONITORING AND CONFIGURATION
───────────────────────────────────────────────────────────────────────────────────

    Dashboard in console: displays the number of events per topic over the last
    24 hours, processing delays, DLQ size.

    Alerts: triggered when DLQ size exceeds 100 messages or processing delay
    exceeds 5 minutes.

    Threshold configuration: the operator can adjust thresholds for T01 (s‑level),
    T02 (automatic application threshold), T08 (budget limits), and others through the
    console interface. Changes are saved in the database and read by agents on next run.

────────────────────────────────────────────────────────────────────────────────────
7.1.4. EXAMPLE OF A FULL TRIGGER CYCLE (T01)
───────────────────────────────────────────────────────────────────────────────────

    WB API Monitor (A2) fetches stock from WB warehouse every 30 minutes and updates
    the sku table.

    After updating, it publishes a stock_updated event to the stock_events topic.

    “Inventory manager” agent (D3) is subscribed to stock_events. On receiving the event:

        For each SKU, checks: stock_total = stock_production + stock_wb.

        Compares against threshold s (calculated by the (s,S) model for each SKU,
        stored in a separate table).

        If stock_total < s, creates a record in production_tasks and publishes a
        stock_low event to the alert_events topic.

    “Production scheduler” agent (L3) is subscribed to alert_events and, upon receiving
    stock_low, adds the task to the production schedule with high priority.

    If the task is not completed (status in_progress) within 2 hours, L3 creates a
    notification for the operator in the console.

7.1.5. SECURITY GUARANTEES
