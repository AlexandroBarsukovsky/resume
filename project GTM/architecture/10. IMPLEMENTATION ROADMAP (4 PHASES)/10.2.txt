**10.2. PHASE 2 (3–4 MONTHS): EXPANSION – STRATEGIST AND DISPATCHER AGENTS**

10.2.1. ACTIONS, RESOURCES, RISKS

**10.2.1.1. Actions**

| Week | Action                                                                  | Responsible                 |
|------|-------------------------------------------------------------------------|-----------------------------|
| 1    | **Architectural design of the strategic layer** – define rules for selecting ideologies (conservative, balanced, aggressive). | RevOps (1), Technical Lead (1) |
| 2–3  | **Develop Strategist agents (3)** – implement in Go, rule engine for selecting ideology based on OKRs and macro indicators, integrate with Kafka. | Engineers (2)                |
| 4    | **Develop Dispatcher agent** – route tasks between tactical groups, load balancing, monitoring. | Engineers (1)                |
| 5–6  | **Expand executor cluster to 800 agents** – add functions: funnel management (800 agents), financial calculations (600), reporting (400). | Engineers (3)                |
| 7    | **Integrate with Gainsight (initial)** – configure Health Score synchronisation, create retention playbook. | RevOps (1)                  |
| 8    | **Test in sandbox** – full run on 500+ accounts, verify distribution by ideologies, correctness of task routing. | Engineers (2), RevOps (1)    |
| 9–10 | **Pilot deployment on 50% of accounts** – monitor for 2 weeks, collect feedback from AEs and CSMs. | RevOps (2), AE (2), CSM (2) |
| 11–12| **Adjust weights and thresholds** – based on pilot data, configure triggers, refine rule engine. | Engineers (1), RevOps (1)    |

**Action Details:**

- **Strategist agents (3):** implemented in Go, deployed on Kubernetes. Each agent is responsible for one ideology:
  * Conservative – priority on retention, low risk, long‑term contracts.
  * Balanced – moderate growth, diversification.
  * Aggressive – maximum expansion, high risk.
  * Input: list of accounts from Level 1, current OKRs (ARR, NRR), macroeconomic indicators (PMI, interest rates, technology index).
  * Output: for each account – selected ideology, saved in the `Strategic_Ideology__c` field in Salesforce via Hightouch.

- **Dispatcher agent:** implemented in Go, subscribes to the Kafka topic `account_strategy`. Aggregates accounts by segment (industry, region, size) and distributes them among tactical groups (Level 3). Ensures load balancing (round‑robin).

- **Expand executor cluster to 800 agents:**
  * New types of executor agents added (in addition to the 200 from Phase 1):
    - 600 agents – financial operations (commission calculation, forecasts, plan‑vs‑actual analysis).
    - 400 agents – reporting (automatic preparation of QBRs, board‑packs, insights).
  * All agents implemented as Lambda functions (fast operations) or containers (long calculations) scaling with Kafka queue.

- **Gainsight integration:**
  * Two‑way Health Score synchronisation (Salesforce ↔ Gainsight) configured.
  * Retention playbook created (trigger: Health Score <0.5) with automatic CSM task creation and email to customer.
  * Expansion playbook (trigger: Expansion Signal) – simplified version for Phase 2.

- **Control panel (extended version):**
  * Added monitoring dashboard for Strategist agents (distribution by ideology, load).
  * Added capability to manually adjust rule engine weights via UI.
  * Improved DLQ handling (ability to resend events, view logs).

**10.2.1.3. Risks and Mitigations**

| Risk                                     | Likelihood | Impact | Mitigation                                                             |
|------------------------------------------|------------|--------|------------------------------------------------------------------------|
| Complexity of calibrating ideology weights | High     | High   | Conduct A/B testing on historical data; use control panel for manual adjustments. |
| Salesforce overload due to increased agent count | Medium | Medium | Use Bulk API for mass updates; implement batch processing.             |
| Gainsight integration instability         | Medium     | Medium | Configure retries (3 attempts), DLQ for events; fallback – create tasks directly in Salesforce. |
| AE resistance to new processes (automatic distribution) | Medium | High   | Provide training, demonstrate improvements in pipeline coverage and reduced routine. |
| Rule engine errors (incorrect ideology selection) | Medium | High   | Log every decision; enable manual override via control panel; monthly retraining. |

4. Rule engine and ideology weights calibrated based on pilot data.
5. Technical documentation for Phase 2 updated, code documented.
