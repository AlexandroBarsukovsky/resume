**8.1. FULL LIST OF TRIGGERS (EVENT → ACTION)**

This protocol details all triggers implemented in the target GTM architecture.
Each trigger describes an event (or combination of events) that initiates one or more
automated actions. Triggers are implemented using Salesforce Platform Events, Apex,
AWS EventBridge, and agents (Level 1–4). All triggers are idempotent and have retry
mechanisms and error handling (Dead Letter Queue).

────────────────────────────────────────────────────────────────────────────────────
**GENERAL TRIGGER STRUCTURE**

Each trigger in the table below is described by the following parameters:
- **ID** – unique identifier.
- **Event** – source and activation condition.
- **Action** – what exactly is performed (specifying the agent or mechanism).
- **Priority** – criticality (High / Medium / Low) affecting processing order.
- **Retry** – number of retry attempts on failure.
- **Error handling** – what happens if the action cannot be performed.

**TECHNICAL IMPLEMENTATION DETAILS**

**Event sources:**
- **Salesforce Platform Events** – for T01, T03–T07, T11–T13 (CRM changes).
- **6sense Webhooks / API** – for T02 (pull requests every 30 minutes).
- **Gainsight Webhooks** – for T03, T04, T06, T10 (Health Score changes, playbooks).
- **Analytic agents** – for T08, T09, T14, T16 (periodic calculations, monitoring).
- **System monitoring** – for T15 (CloudWatch / Datadog alerts).

**Action executors:**
- **Executor agents (Level 4)** – create tasks in Salesforce, call APIs (SalesLoft, Gainsight, Ironclad).
- **Analytic agents (Level 1)** – recalculate scoring, aggregate metrics.
- **Financial agents (Level 4)** – calculate forecasts, headcount, quotas.
- **Apex (Salesforce)** – publish Platform Events, triggers on objects.
- **AWS EventBridge** – route events between systems.

**Error handling:**
- All triggers have a **Dead Letter Queue (DLQ)** where events that fail after the specified number of attempts are placed.
- RevOps operators receive DLQ notifications via the control panel and can manually perform the action or restart processing.
- For critical triggers (T01, T03, T04, T06, T10), alerts are configured in PagerDuty.

────────────────────────────────────────────────────────────────────────────────────
**EXAMPLE PROCESS (T01)**

1. **Event:** An analytic agent recalculates Account Score for an account, result >0.7.
2. **Publication:** The agent updates the `Account_Score__c` field in Salesforce via Hightouch.
   A Salesforce Apex trigger on the Account object publishes the Platform Event `Account_Change__e`.
3. **Routing:** EventBridge forwards the event to Kafka (topic `trigger_events`).
4. **Processing:** An executor agent (Lambda) subscribed to the topic receives the event.
5. **Actions:**
   - Call Salesforce API to create a task (Task) of type `Sales_Strategist_Task`.
   - Call SalesLoft API to add the account to a cadence.
6. **Logging:** Success/failure are recorded in CloudWatch; on failure – retry after 10, 60, 300 seconds.
7. **DLQ:** If the error persists after 3 attempts, the event is placed in SQS (DLQ), and a RevOps operator is notified.

**CONCLUSIONS ON TRIGGERS**

- The presented list of 16 triggers covers all key events of GTM processes,
  ensuring full automation of responses.

  agents, and external services, providing scalability and fault tolerance.
